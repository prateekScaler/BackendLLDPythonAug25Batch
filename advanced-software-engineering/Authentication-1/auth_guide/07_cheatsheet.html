üîÑ Encoding vs Encryption vs Hashing

Transform Reversible? Key Needed? Use For
Encoding Yes, by anyone No Format conversion (Base64, URL)
Encryption Yes, with key Yes Confidentiality (AES, RSA)
Hashing No, never No Verification (BCrypt, SHA)
Remember: Base64 is NOT security! Anyone can decode it.

üç™ Session-Based vs Token-Based Auth

Aspect Session-Based Token-Based (JWT)
Server stores Session data Nothing (stateless)
Client stores Session ID (cookie) JWT token
Scaling Needs shared store (Redis) Easy (no shared state)
Revocation Easy (delete session) Hard (token is self-contained)
Best for Traditional web apps APIs, mobile, microservices

‚úÖ Security Rules

DO

  • Use BCrypt/Argon2 for passwords
  • Use cost factor >= 12
  • Use HTTPS always
  • Set HttpOnly on cookies
  • Implement session expiry
  • Regenerate session ID after login

DON'T

  • Store plaintext passwords
  • Use MD5 or unsalted SHA
  • Think Base64 = encryption
  • Store sensitive data in cookies
  • Skip CSRF protection
  • Use predictable session IDs

Golden Rules Summary

  1. Authentication != Authorization - 401 vs 403
  2. Never store plaintext passwords - Use BCrypt (cost >= 12)
  3. Encoding != Encryption - Base64 is NOT security
  4. Session-based for most web apps - Token-based for APIs
  5. Always verify on backend - Never trust frontend alone
  6. Use RBAC for permissions - Never hardcode role checks
